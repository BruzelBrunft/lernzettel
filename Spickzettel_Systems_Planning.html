F04 Anforderungsanalyse
▼
User- vs. Systemanforderungen
User-AnforderungenSystemanforderungen
CharakterNatürliche Sprache, High-LevelDetailliert, technisch
Für wen?Manager ohne TechnikwissenEntwickler
DokumentLastenheftPflichtenheft
Funktional vs. Nicht-funktional
Funktional = WAS das System tun soll (z.B. User kann Geld überweisen, Konto erstellen)
Nicht-funktional = WIE GUT (z.B. Ladezeit < 2 Sek., Security, Usability)
Schnelle Antwortzeiten = NICHT-funktional! Konto erstellen = funktional!
Nicht-funktionale Anforderungen nicht erfüllt → GANZES System abgelehnt!

Externe Anforderungen = Gesetze, ethische Vorgaben

Klassendiagramm
Aufbau: Oben = Name · Mitte = Attribute · Unten = Operationen
Generalisierung/Vererbung = Pfeil zur Elternklasse

Komposition = gefüllte Raute (Teil kann NICHT ohne Ganzes existieren)

Interaktionsmodelle
Use Case = WER macht WAS mit dem System (Strichfigur → Ellipse → Rechteck)
Sequenzdiagramm = zeitliche Abfolge von oben nach unten, Lebenslinien
Verhaltensmodelle
Aktivitätsdiagramm = datengetrieben, Daten fließen durch Schritte (gefüllter Kreis = Start)
Zustandsdiagramm = ereignisgetrieben, Stimuli lösen Zustandswechsel aus, Superstates
Zustandsdiagramm zeigt KEINEN Datenfluss — nur Verhalten!

F07 Soziotechnische Systeme
▼
Technisch vs. Soziotechnisch
Technisch = Hardware + Software, KEINE Menschen/Prozesse (z.B. Fernseher, PC)
Soziotechnisch = + Menschen + Prozesse + Organisation (z.B. Krankenhaus, Buchverlag)
Fernseher/PC = technisch! Krankenhaus/Buchverlag = soziotechnisch!
7 Schichten
Harte Burschen Können Programmieren, Gibt Ordentliche Software
Hardware → Betriebssystem → Kommunikation & Daten → Programme → Geschäftsprozesse → Organisation → Gesellschaft
Interaktionen meist zwischen Nachbarschichten — aber nicht immer (z.B. Gesetz → direkt auf Programm)
3 Merkmale soziotechnischer Systeme
ENO — Emergent · Non-deterministisch · Organisatorisch
Emergente Eigenschaften = entstehen erst aus dem Zusammenspiel aller Teile
Non-deterministisch = bei gleicher Eingabe nicht immer dasselbe Ergebnis (Menschen!)
Organisatorisch zielabhängig = Erfolg hängt von Managementzielen ab
Soziotechn. Systeme liefern bei gleicher Eingabe NICHT immer dasselbe!
3 Organisatorische Einflussfaktoren
PJO — Prozess · Job · Organisation
Prozessänderungen = neue Arbeitsabläufe → Widerstand
Jobveränderung = Dequalifizierung → aktiver Widerstand
Organisatorische Änderungen = veränderte Machtstrukturen
3 Hauptprozesse Systems Engineering
PDO — Procurement · Development · Operation
Procurement (Beschaffung) = was kaufen wir? COTS = Standardprodukte von der Stange
Development (Entwicklung) = Anforderungsanalyse als Brücke, Systemintegration schwierig
Operation (Betrieb) = Schulung, unentdeckte Fehler, Schnittstellenprobleme
COTS entsprechen SELTEN genau den Anforderungen!

Pricing to win = Preis fix, Umfang variabel
Kostenschätzung
2 Methoden: Erfahrungsbasiert (Expertenwissen) vs. Algorithmisch (Formel)
Aufwand = A × Größe^B × M
A = organisationsabhängig · B = 1–1,5 (exponentiell!) · M = Multiplikator
Aufwand (Personenmonate) = größter Kostenposten!
Mehr Leute ≠ schneller! Mehr Kommunikations-Overhead!
Preis ≠ einfach Kosten + 10% Gewinn — viele Faktoren spielen rein!

BegriffBedeutung

Organisatorische Ebene = Standards für Doku, Design, Implementierung

Softwarequalität
Qualitätsprobleme erstmals in den 1960er Jahren erkannt
Qualität betrifft hauptsächlich nichtfunktionale Anforderungen
15 Qualitätsmerkmale (Boehm et al.) — nicht alle gleichzeitig optimierbar!
Software wird entwickelt, nicht hergestellt → Toleranzbegriff der Fertigung greift nicht
Softwarequalität ist SUBJEKTIV — nie 'objektiv' ankreuzen!
Standards
Produktstandards = Dokumentations-, Kodierungsstandards
Prozessstandards = Spezifikations-, Design-, Validierungsprozesse
ISO 9001 = internationales Normenwerk, Qualitätshandbuch, ursprünglich 1987
ISO-9001-Zertifizierung garantiert NICHT bessere Software!
Reviews & Inspektionen
3 Phasen: Pre-Review → Review-Meeting (~2h) → Post-Review
Review-Team: 3–4 Personen
Ziel: Qualitätsverbesserung — NICHT Mitarbeiterbewertung!
Software-Inspektionen = Peer-Reviews, zeilenweise Code-Überprüfung
Agil: Scrum = Review je Iteration · XP = Pair Programming als laufende Überprüfung
